## 1. Why This Exists

In earlier chapters, MiniURLShortener became a distributed system:

```text
Client
  |
  v
Spring Cloud Gateway
  |
  v
URL Service
  |
  +--> Postgres
  +--> Redis
  +--> Kafka
```

But real backend systems rarely live as one service.

A URL shortener may need to call other internal services:

```text
Auth Service        -> validate user / API key
Analytics Service   -> fetch click summaries
Abuse Service       -> check phishing/malware risk
Billing Service     -> check plan limits
Notification Service-> send alerts
User Service        -> fetch user profile
```

Without OpenFeign, service-to-service calls often become noisy:

```java
RestTemplate / WebClient code
manual URL building
manual JSON mapping
manual error handling
manual headers
manual retry/circuit breaker wiring
```

OpenFeign gives a cleaner programming model:

```java
@FeignClient(name = "abuse-service")
public interface AbuseClient {

    @PostMapping("/internal/v1/abuse/check")
    AbuseCheckResponse check(@RequestBody AbuseCheckRequest request);
}
```

Then service code can call:

```java
AbuseCheckResponse response = abuseClient.check(request);
```

But the senior mindset is critical:

```text
It looks like a Java method call.
It behaves like a remote HTTP call.
```

That means it can fail due to:

```text
network timeout
DNS/service discovery issue
remote 500
remote 404/403/429
serialization mismatch
bad contract version
slow downstream
retry storm
thread pool exhaustion
circuit breaker open
```

Production mental model:

```text
Feign improves developer ergonomics.
It does not remove distributed-system failure.
```

## 2. The One Core Mental Model

OpenFeign is:

```text
A DECLARATIVE HTTP CLIENT
```

You declare the remote API as a Java interface.
Spring creates a proxy implementation at runtime.
When you call the method, the proxy performs an HTTP request.

ASCII:

```text
Your Service Code
      |
      | Java method call
      v
+---------------------+
| Feign Proxy         |
| generated by Spring |
+---------------------+
      |
      | HTTP request
      v
+---------------------+
| Remote Service      |
| real network call   |
+---------------------+
      |
      | HTTP response
      v
Your Service Code
```

One-line memory:

```text
Feign makes remote HTTP look local, but you must design it as remote.
```

Wrong mental model:

```text
Feign is just like calling another Spring service bean.
```

Correct mental model:

```text
Feign is a network boundary hidden behind a Java interface.
```

This difference decides whether your production system survives.

## 4. Where OpenFeign Fits In Spring Cloud

Spring Cloud commonly has these pieces:

```text
Eureka              -> service registry
Spring Cloud Gateway-> entry point / routing
OpenFeign           -> service-to-service HTTP client
Resilience4j        -> circuit breaker / retry / timeout protection
Micrometer          -> metrics
Sleuth/Micrometer Tracing -> trace propagation
```

ASCII:

```text
External Client
      |
      v
+----------------------+        +------------------+
| Spring Cloud Gateway | -----> | url-service      |
+----------------------+        +------------------+
                                      |
                                      | Feign call
                                      v
                               +------------------+
                               | abuse-service    |
                               +------------------+
```

Role separation:

```text
Gateway:
    handles external routing into the system

Eureka:
    tells services where other services are

Feign:
    makes one service call another service

Resilience4j:
    protects the caller from downstream failure
```

Senior distinction:

```text
Gateway is north-south traffic.
Feign is east-west traffic.
```

Meaning:

```text
North-south:
    client -> backend system

East-west:
    service -> service inside backend system
```

## 5. Feign Is Not Magic

Feign hides HTTP boilerplate, not HTTP reality.

Local method call:

```text
nanoseconds/microseconds
same process
same memory
no network
failure usually exception/bug
```

Feign call:

```text
milliseconds/seconds
remote process
network boundary
serialization/deserialization
HTTP status codes
timeouts
retries
partial failures
```

ASCII:

```text
Looks like:

urlSafetyChecker.check(url)

Actually:

serialize JSON
  -> resolve service instance
  -> open connection
  -> send HTTP request
  -> wait for remote service
  -> receive response
  -> deserialize JSON
  -> map errors/exceptions
```

Danger:

```java
public void createUrl(CreateShortUrlRequest request) {
    abuseClient.check(request.getLongUrl());
    repository.save(...);
}
```

This looks simple.

But if Abuse Service is slow, URL Service threads can pile up.

ASCII failure:

```text
URL Service threads
  |
  +-- waiting for Abuse Service
  +-- waiting for Abuse Service
  +-- waiting for Abuse Service
  +-- waiting for Abuse Service
  +-- waiting for Abuse Service

Incoming requests continue...

Result:
URL Service outage caused by Abuse Service slowness.
```

Production rule:

```text
Every Feign call needs timeout, error mapping, observability, and resilience thinking.
```

## 6. URL Shortener Use Case

We will use this internal service call:

```text
url-service -> abuse-service
```

Question:

```text
Can this longUrl be shortened?
```

Possible responses:

```text
SAFE
SUSPICIOUS
MALICIOUS
UNKNOWN
```

Business behavior:

```text
SAFE        -> allow
SUSPICIOUS  -> allow or manual review depending product policy
MALICIOUS   -> reject
UNKNOWN     -> maybe reject or allow with flag
```

For this chapter:

```text
SAFE      -> create short URL
MALICIOUS -> reject with 403 URL_BLOCKED_BY_ABUSE_POLICY
TIMEOUT   -> reject with 503 ABUSE_SERVICE_UNAVAILABLE
```

Why fail closed?

A URL shortener is abuse-sensitive.
If the safety service is unavailable, allowing everything may create phishing/spam risk.

Tradeoff:

```text
Fail open:
    better availability
    worse security

Fail closed:
    worse availability
    better security
```

For high-risk security checks:

```text
fail closed is often safer
```

For non-critical metadata calls:

```text
fail open is often acceptable
```

## 7. Synchronous Service-To-Service Flow

OpenFeign is usually synchronous.

Flow:

```text
Client request waits while URL Service calls Abuse Service.
```

ASCII:

```text
Client
  |
  | POST /api/v1/urls
  v
URL Service
  |
  | POST /internal/v1/abuse/check
  v
Abuse Service
  |
  | response SAFE/MALICIOUS
  v
URL Service
  |
  | create or reject
  v
Client
```

Latency math:

```text
URL create without Feign:
    validation 5ms
    DB insert 15ms
    total 20ms

URL create with Feign:
    validation 5ms
    abuse call 80ms
    DB insert 15ms
    total 100ms
```

One downstream call can dominate latency.

At scale:

```text
100 RPS and 80ms downstream latency
rough concurrent waiting calls = RPS * latency_seconds
= 100 * 0.08
= 8 waiting calls

10,000 RPS and 80ms
= 10,000 * 0.08
= 800 waiting calls
```

That is why timeouts and bulkheads matter.

## 9. Enable Feign Clients

Main application:

```java
package com.miniurl.shortener;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.openfeign.EnableFeignClients;

@SpringBootApplication
@EnableFeignClients
public class UrlServiceApplication {

    public static void main(String[] args) {
        SpringApplication.run(UrlServiceApplication.class, args);
    }
}
```

What `@EnableFeignClients` does:

```text
1. Scans for interfaces annotated with @FeignClient.
2. Creates runtime proxy beans.
3. Registers them in Spring ApplicationContext.
4. Allows services to inject them like normal beans.
```

ASCII:

```text
Startup
  |
  v
@EnableFeignClients
  |
  v
scan @FeignClient interfaces
  |
  v
create proxy bean
  |
  v
inject into service
```

Common mistake:

```text
Feign client interface is outside component scan package.
```

Fix:

```java
@EnableFeignClients(basePackages = "com.miniurl.shortener")
```

## 10. Create The Remote DTO Contract

Feign client and remote service must agree on request/response JSON.

Package suggestion:

```text
com.miniurl.shortener.integration.abuse
```

Request DTO:

```java
package com.miniurl.shortener.integration.abuse;

public class AbuseCheckRequest {

    private String longUrl;
    private String userId;
    private String source;

    public AbuseCheckRequest() {
    }

    public AbuseCheckRequest(String longUrl, String userId, String source) {
        this.longUrl = longUrl;
        this.userId = userId;
        this.source = source;
    }

    public String getLongUrl() {
        return longUrl;
    }

    public void setLongUrl(String longUrl) {
        this.longUrl = longUrl;
    }

    public String getUserId() {
        return userId;
    }

    public void setUserId(String userId) {
        this.userId = userId;
    }

    public String getSource() {
        return source;
    }

    public void setSource(String source) {
        this.source = source;
    }
}
```

Response DTO:

```java
package com.miniurl.shortener.integration.abuse;

public class AbuseCheckResponse {

    private String verdict;
    private String reasonCode;
    private int riskScore;

    public AbuseCheckResponse() {
    }

    public String getVerdict() {
        return verdict;
    }

    public void setVerdict(String verdict) {
        this.verdict = verdict;
    }

    public String getReasonCode() {
        return reasonCode;
    }

    public void setReasonCode(String reasonCode) {
        this.reasonCode = reasonCode;
    }

    public int getRiskScore() {
        return riskScore;
    }

    public void setRiskScore(int riskScore) {
        this.riskScore = riskScore;
    }

    public boolean isMalicious() {
        return "MALICIOUS".equalsIgnoreCase(verdict);
    }
}
```

Example JSON response:

```json
{
  "verdict": "SAFE",
  "reasonCode": "NO_MATCH",
  "riskScore": 5
}
```

Contract rule:

```text
Feign DTOs are API contracts.
Treat them carefully like public DTOs.
```

Avoid exposing internal entity classes through Feign.

Wrong:

```java
AbuseCheckResponse check(ShortUrlEntity entity);
```

Correct:

```java
AbuseCheckResponse check(AbuseCheckRequest request);
```

## 11. Create The Feign Client Interface

Client:

```java
package com.miniurl.shortener.integration.abuse;

import org.springframework.cloud.openfeign.FeignClient;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;

@FeignClient(
        name = "abuse-service",
        path = "/internal/v1/abuse",
        configuration = AbuseFeignConfig.class
)
public interface AbuseClient {

    @PostMapping("/check")
    AbuseCheckResponse checkUrl(@RequestBody AbuseCheckRequest request);
}
```

Meaning:

```text
name = abuse-service
    service id used for Eureka/load balancing

path = /internal/v1/abuse
    base path of remote controller

@PostMapping('/check')
    endpoint path
```

Full remote URL logically becomes:

```text
http://abuse-service/internal/v1/abuse/check
```

If Eureka is enabled:

```text
abuse-service resolves to one real instance
```

If no Eureka:

```text
configure url explicitly
```

Example explicit URL:

```java
@FeignClient(
        name = "abuse-service",
        url = "${clients.abuse-service.url}",
        path = "/internal/v1/abuse"
)
public interface AbuseClient {
    @PostMapping("/check")
    AbuseCheckResponse checkUrl(@RequestBody AbuseCheckRequest request);
}
```

application.yml:

```yaml
clients:
  abuse-service:
    url: http://localhost:8082
```

Rule:

```text
Use name with Eureka in cloud.
Use url for local/simple environments.
```

## 12. Service Layer Integration

Create domain exception:

```java
package com.miniurl.shortener.url.service;

import com.miniurl.shortener.common.error.ApiException;
import org.springframework.http.HttpStatus;

public class UrlBlockedByAbusePolicyException extends ApiException {

    public UrlBlockedByAbusePolicyException() {
        super(
                "URL_BLOCKED_BY_ABUSE_POLICY",
                HttpStatus.FORBIDDEN,
                "URL is blocked by abuse policy"
        );
    }
}
```

Downstream unavailable exception:

```java
package com.miniurl.shortener.url.service;

import com.miniurl.shortener.common.error.ApiException;
import org.springframework.http.HttpStatus;

public class AbuseServiceUnavailableException extends ApiException {

    public AbuseServiceUnavailableException() {
        super(
                "ABUSE_SERVICE_UNAVAILABLE",
                HttpStatus.SERVICE_UNAVAILABLE,
                "URL safety check is temporarily unavailable"
        );
    }
}
```

Service:

```java
package com.miniurl.shortener.url.service;

import com.miniurl.shortener.integration.abuse.AbuseCheckRequest;
import com.miniurl.shortener.integration.abuse.AbuseCheckResponse;
import com.miniurl.shortener.integration.abuse.AbuseClient;
import org.springframework.stereotype.Service;

@Service
public class UrlSafetyService {

    private final AbuseClient abuseClient;

    public UrlSafetyService(AbuseClient abuseClient) {
        this.abuseClient = abuseClient;
    }

    public void assertUrlIsSafe(String longUrl, String userId) {
        AbuseCheckRequest request = new AbuseCheckRequest(
                longUrl,
                userId,
                "MINI_URL_SHORTENER"
        );

        AbuseCheckResponse response = abuseClient.checkUrl(request);

        if (response == null) {
            throw new AbuseServiceUnavailableException();
        }

        if (response.isMalicious()) {
            throw new UrlBlockedByAbusePolicyException();
        }
    }
}
```

Use in create flow:

```java
@Service
public class ShortUrlService {

    private final UrlValidatorService validatorService;
    private final UrlSafetyService urlSafetyService;
    private final ShortUrlRepository repository;

    public ShortUrlService(
            UrlValidatorService validatorService,
            UrlSafetyService urlSafetyService,
            ShortUrlRepository repository
    ) {
        this.validatorService = validatorService;
        this.urlSafetyService = urlSafetyService;
        this.repository = repository;
    }

    public CreateShortUrlResponse create(CreateShortUrlCommand command) {
        validatorService.validateLongUrl(command.longUrl());

        urlSafetyService.assertUrlIsSafe(
                command.longUrl(),
                command.userId()
        );

        // generate short code, save entity, return response
        return createAndPersist(command);
    }
}
```

Design rule:

```text
Controller should not call Feign directly.
Service should own business meaning of remote call.
```

## 13. Controller Flow With Feign

Controller remains clean:

```java
@PostMapping("/api/v1/urls")
public ResponseEntity<CreateShortUrlResponse> create(
        @Valid @RequestBody CreateShortUrlRequest request
) {
    CreateShortUrlCommand command = new CreateShortUrlCommand(
            request.getLongUrl(),
            request.getCustomAlias(),
            request.getExpiresAt(),
            getCurrentUserId()
    );

    CreateShortUrlResponse response = shortUrlService.create(command);

    return ResponseEntity.status(HttpStatus.CREATED).body(response);
}
```

ASCII:

```text
Controller
  |
  | DTO validation
  v
ShortUrlService
  |
  | business validation
  v
UrlSafetyService
  |
  | Feign call
  v
Abuse Service
  |
  | verdict
  v
ShortUrlService
  |
  | save or reject
  v
Controller response
```

Why this is clean:

```text
Controller handles HTTP request/response.
ShortUrlService handles create business flow.
UrlSafetyService hides downstream integration.
AbuseClient hides HTTP details.
```

## 14. Feign With Eureka Service Discovery

With Eureka, services register themselves:

```text
url-service registers
abuse-service registers
analytics-service registers
```

Feign uses service name:

```java
@FeignClient(name = "abuse-service")
```

Runtime flow:

```text
1. URL Service calls abuseClient.checkUrl().
2. Feign asks LoadBalancer for abuse-service instance.
3. LoadBalancer uses service registry data.
4. One abuse-service instance is selected.
5. HTTP call is made to that instance.
```

ASCII:

```text
URL Service
  |
  | call abuse-service
  v
Spring Cloud LoadBalancer
  |
  | find instances
  v
Eureka Registry
  |
  | instance list
  v
abuse-service instance A/B/C
```

Example:

```text
abuse-service instances:
    10.0.1.11:8082
    10.0.1.12:8082
    10.0.1.13:8082

Feign call chooses one.
```

Benefit:

```text
No hard-coded host/port in app code.
```

Danger:

```text
If service name is wrong, discovery fails.
```

Common error:

```text
No instances available for abuse-service
```

Debug:

```text
Check Eureka dashboard.
Check spring.application.name of Abuse Service.
Check @FeignClient(name = ...).
```

## 15. Feign With Spring Cloud Gateway

There are two possible internal call styles.

### Style 1: Service-to-service direct through discovery

```text
url-service -> abuse-service
```

This is common for internal microservice calls.

### Style 2: Service calls through Gateway

```text
url-service -> gateway -> abuse-service
```

This can centralize auth/routing, but adds extra hop and couples internal calls to gateway availability.

ASCII comparison:

```text
Direct:

url-service ------Feign------> abuse-service

Through gateway:

url-service ------Feign------> gateway ------route------> abuse-service
```

For MiniURLShortener:

```text
Prefer direct service discovery for internal trusted service-to-service calls.
Use gateway for external client ingress.
```

When gateway internal calls make sense:

```text
1. Need central policy enforcement.
2. Need a single internal API endpoint.
3. Need cross-cluster routing.
```

When direct Feign is better:

```text
1. Lower latency.
2. Less dependency on gateway.
3. Simpler internal topology.
```

Senior answer:

```text
Gateway is not mandatory for every internal service call. Feign can call services directly using discovery and load balancing.
```

## 16. Timeout Configuration

Default timeout choices are often unsafe.

Set explicit connect and read timeouts.

application.yml:

```yaml
spring:
  cloud:
    openfeign:
      client:
        config:
          abuse-service:
            connectTimeout: 500
            readTimeout: 1000
```

Meaning:

```text
connectTimeout:
    time allowed to establish connection

readTimeout:
    time allowed waiting for response data
```

ASCII:

```text
URL Service
  |
  | connect timeout 500ms
  v
Abuse Service
  |
  | read timeout 1000ms
  v
Response
```

Why short timeouts matter:

```text
If Abuse Service hangs for 30 seconds, URL Service request threads block.
Short timeout fails fast and protects caller.
```

Bad:

```text
No timeout.
Request waits too long.
Thread pool fills.
URL Service fails globally.
```

Better:

```text
Timeout quickly.
Map to controlled 503.
Circuit breaker opens if repeated.
```

Rule of thumb:

```text
Set downstream timeout lower than your API SLA.
```

Example:

```text
Create URL SLA p99 = 300ms
Abuse call timeout = 100ms-150ms if strict
```

But if Abuse Service needs deeper scanning, make it async or use cached verdicts.

## 17. Error Decoder

Remote service may return HTTP errors:

```text
400 bad request
403 malicious URL
429 rate limited
500 internal error
503 unavailable
```

Feign can decode these into meaningful exceptions.

Config:

```java
package com.miniurl.shortener.integration.abuse;

import feign.Response;
import feign.codec.ErrorDecoder;
import org.springframework.context.annotation.Bean;

public class AbuseFeignConfig {

    @Bean
    public ErrorDecoder abuseErrorDecoder() {
        return new AbuseErrorDecoder();
    }
}
```

Decoder:

```java
package com.miniurl.shortener.integration.abuse;

import com.miniurl.shortener.url.service.AbuseServiceUnavailableException;
import com.miniurl.shortener.url.service.UrlBlockedByAbusePolicyException;
import feign.Response;
import feign.codec.ErrorDecoder;

public class AbuseErrorDecoder implements ErrorDecoder {

    private final ErrorDecoder defaultDecoder = new Default();

    @Override
    public Exception decode(String methodKey, Response response) {
        int status = response.status();

        if (status == 403) {
            return new UrlBlockedByAbusePolicyException();
        }

        if (status == 429 || status == 500 || status == 502 || status == 503 || status == 504) {
            return new AbuseServiceUnavailableException();
        }

        return defaultDecoder.decode(methodKey, response);
    }
}
```

Why this matters:

```text
Remote HTTP errors should become local domain/integration exceptions.
```

Without decoder:

```text
FeignException$Forbidden
FeignException$InternalServerError
```

With decoder:

```text
URL_BLOCKED_BY_ABUSE_POLICY
ABUSE_SERVICE_UNAVAILABLE
```

This keeps your global exception handler consistent.

## 18. Retry Policy

Retries are dangerous and useful.

Useful for:

```text
temporary network glitch
connection reset
short 503 spike
```

Dangerous for:

```text
non-idempotent POST
remote overload
long-running requests
rate-limited downstream
```

ASCII retry storm:

```text
Original load: 1000 RPS
Retry once:    2000 calls possible
Retry twice:   3000 calls possible

Downstream already unhealthy
Retries make it worse
```

For abuse check:

```text
POST /check is logically read-only.
It does not create business state.
So limited retry may be acceptable.
```

But still keep it small:

```text
max attempts: 1 or 2
small backoff
only retry transient failures
```

Simple Feign Retryer config:

```java
package com.miniurl.shortener.integration.abuse;

import feign.Retryer;
import org.springframework.context.annotation.Bean;

public class AbuseFeignConfig {

    @Bean
    public Retryer retryer() {
        return new Retryer.Default(
                100,  // initial interval ms
                300,  // max interval ms
                2     // max attempts including first call
        );
    }
}
```

Production rule:

```text
Retry only when the operation is idempotent or safely repeatable.
```

For create payment/order APIs:

```text
Never blindly retry without idempotency key.
```

For URL safety check:

```text
Limited retry is acceptable if the remote check is side-effect free.
```

## 19. Circuit Breaker Integration

A circuit breaker prevents repeated calls to a failing downstream.

Mental model:

```text
If Abuse Service keeps failing, stop calling it temporarily.
Fail fast instead of waiting for timeout every time.
```

ASCII:

```text
CLOSED
  |
  | failures exceed threshold
  v
OPEN
  |
  | wait duration
  v
HALF_OPEN
  |
  | test calls succeed
  v
CLOSED
```

Add dependency usually through Spring Cloud CircuitBreaker + Resilience4j.

application.yml example:

```yaml
spring:
  cloud:
    openfeign:
      circuitbreaker:
        enabled: true

resilience4j:
  circuitbreaker:
    instances:
      abuse-service:
        slidingWindowSize: 20
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3
```

Why this matters:

```text
Without circuit breaker:
    each request waits for failure

With circuit breaker:
    after repeated failures, requests fail fast
```

For URL shortener:

```text
If abuse-service is down, do not let create-url endpoint slowly die.
Return controlled 503 quickly.
```

## 20. Fallback Design

Fallback is what happens when Feign call fails.

Bad fallback:

```text
Always return SAFE if Abuse Service fails.
```

This may allow malicious URLs.

Better fallback for security-critical call:

```text
Throw AbuseServiceUnavailableException.
```

Fallback factory:

```java
package com.miniurl.shortener.integration.abuse;

import com.miniurl.shortener.url.service.AbuseServiceUnavailableException;
import org.springframework.cloud.openfeign.FallbackFactory;
import org.springframework.stereotype.Component;

@Component
public class AbuseClientFallbackFactory implements FallbackFactory<AbuseClient> {

    @Override
    public AbuseClient create(Throwable cause) {
        return request -> {
            throw new AbuseServiceUnavailableException();
        };
    }
}
```

Feign client:

```java
@FeignClient(
        name = "abuse-service",
        path = "/internal/v1/abuse",
        configuration = AbuseFeignConfig.class,
        fallbackFactory = AbuseClientFallbackFactory.class
)
public interface AbuseClient {

    @PostMapping("/check")
    AbuseCheckResponse checkUrl(@RequestBody AbuseCheckRequest request);
}
```

Fallback decision table:

```text
+----------------------+------------------------+---------------------+
| Remote Call          | Criticality            | Fallback            |
+----------------------+------------------------+---------------------+
| abuse check          | security critical      | fail closed 503     |
| analytics summary    | non-critical           | empty/default data  |
| notification send    | can be async/retried   | enqueue retry       |
| profile display      | maybe non-critical     | cached profile      |
+----------------------+------------------------+---------------------+
```

Senior rule:

```text
Fallback is a business decision, not only technical behavior.
```

## 21. Headers And Correlation ID

Distributed calls need traceability.

Client request may enter with:

```text
X-Correlation-Id: req-123
```

URL Service should pass it to Abuse Service.

ASCII:

```text
Client
  |
  | X-Correlation-Id: req-123
  v
URL Service logs req-123
  |
  | Feign forwards req-123
  v
Abuse Service logs req-123
```

Feign RequestInterceptor:

```java
package com.miniurl.shortener.integration.common;

import feign.RequestInterceptor;
import jakarta.servlet.http.HttpServletRequest;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.context.request.RequestContextHolder;
import org.springframework.web.context.request.ServletRequestAttributes;

@Configuration
public class FeignHeaderPropagationConfig {

    @Bean
    public RequestInterceptor correlationIdInterceptor() {
        return template -> {
            ServletRequestAttributes attributes =
                    (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();

            if (attributes == null) {
                return;
            }

            HttpServletRequest request = attributes.getRequest();
            String correlationId = request.getHeader("X-Correlation-Id");

            if (correlationId != null && !correlationId.isBlank()) {
                template.header("X-Correlation-Id", correlationId);
            }
        };
    }
}
```

Important:

```text
Propagate only safe headers.
Do not blindly forward all incoming headers.
```

Safe examples:

```text
X-Correlation-Id
X-Request-Id
traceparent
```

Dangerous to blindly forward:

```text
Authorization
Cookie
internal debug headers
user-controlled spoofing headers
```

## 22. Authentication Between Services

Internal services should not be completely open.

Possible strategies:

```text
1. Network-level trust only.
2. Static internal API key.
3. mTLS.
4. OAuth2 client credentials.
5. JWT token exchange.
```

For MiniURLShortener learning version, simple internal API key is enough.

application.yml:

```yaml
internal:
  service-api-key: ${INTERNAL_SERVICE_API_KEY}
```

Interceptor:

```java
package com.miniurl.shortener.integration.common;

import feign.RequestInterceptor;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class FeignAuthConfig {

    @Bean
```

```text
X-Internal-Api-Key must match configured secret
```

Production note:

```text
For serious production, prefer mTLS or OAuth2 client credentials over static secrets.
```

Security rule:

```text
Do not log internal API keys.
Do not commit secrets in Git.
Rotate secrets.
```

### Unit test service behavior

Mock AbuseClient:

```java
@ExtendWith(MockitoExtension.class)
class UrlSafetyServiceTest {

    @Mock
    AbuseClient abuseClient;

    @InjectMocks
    UrlSafetyService urlSafetyService;

    @Test

    @Test

                UrlBlockedByAbusePolicyException.class,
```

WireMock can simulate remote HTTP service.

### Dry Run 3: Abuse Service Timeout


```text

URL Service <--- no response within timeout
URL Service ---- 503 -------> Client
```

```text
The URL Service itself did not crash.

503 is more meaningful.
```

## 27. Production Failure Stories

### Failure Story 1: No timeout caused thread exhaustion

Abuse Service became slow.
URL Service Feign calls waited too long.
Tomcat threads piled up.
Create URL API became unavailable.

Root cause:

```text
No explicit Feign timeout.
```

Fix:

```text
Set connectTimeout/readTimeout.
Add circuit breaker.
Fail fast with 503.
```

Lesson:

```text
Downstream slowness can kill the caller.
```

---

### Failure Story 2: Retry storm during outage

Abuse Service started returning 503.
URL Service retried each request three times.
Traffic to Abuse Service tripled.
Recovery became slower.

Root cause:

```text
Aggressive retries without circuit breaker.
```

Fix:

```text
Limit retries.
Use backoff.
Use circuit breaker.
Retry only safe operations.
```

Lesson:

```text
Retries are load multipliers.
```

---

### Failure Story 3: Contract mismatch broke production

Abuse Service renamed JSON field:

```text
verdict -> status
```

URL Service still expected:

```text
verdict
```

Feign deserialized verdict as null.
URL Service allowed unsafe behavior or failed unexpectedly.

Root cause:

```text
No contract test/versioning.
```

Fix:

```text
Consumer-driven contract tests.
Backward-compatible DTO changes.
Strict null handling.
```

Lesson:

```text
Feign interfaces are contracts, not private implementation details.
```

---

### Failure Story 4: Full Feign logging leaked secrets

Production enabled FULL Feign logging.
Long URLs contained reset tokens.
Logs exposed sensitive data.

Root cause:

```text
Unsafe logging level and no redaction.
```

Fix:

```text
Use BASIC in production.
Redact sensitive query params.
Do not log full longUrl.
```

Lesson:

```text
Integration logs can become data leaks.
```

---

### Failure Story 5: Fallback allowed malicious URLs

Fallback returned SAFE when Abuse Service was down.
Attackers noticed and shortened malicious URLs during outage.

Root cause:

```text
Fail-open fallback for security-critical dependency.
```

Fix:

```text
Fail closed for abuse checks.
Return controlled 503.
```

Lesson:

```text
Fallback must match business risk.
```

## 29. Common Mistakes

### Mistake 1: Treating Feign like local service call

Wrong:

```text
No timeout, no error mapping, no fallback, no metrics.
```

Correct:

```text
Design every Feign call as remote and failure-prone.
```

### Mistake 2: Calling Feign from controller

Wrong:

```java
controller -> abuseClient
```

Correct:

```java
controller -> service -> integration service -> abuseClient
```

### Mistake 3: No explicit timeout

Wrong:

```text
Use defaults blindly.
```

Correct:

```text
Set connectTimeout and readTimeout per client.
```

### Mistake 4: Blind retries

Wrong:

```text
Retry all methods and all statuses.
```

Correct:

```text
Retry only transient failures and safe/idempotent calls.
```

### Mistake 5: Bad fallback

Wrong:

```text
Fallback always returns success.
```

Correct:

```text
Fallback reflects business criticality.
```

### Mistake 6: Sharing entity classes

Wrong:

```text
Feign request/response uses JPA entity.
```

Correct:

```text
Use stable DTO contracts.
```

### Mistake 7: FULL logging in production

Wrong:

```text
Log headers and bodies everywhere.
```

Correct:

```text
Use safe structured logging and redaction.
```

### Mistake 8: Ignoring versioning

Wrong:

```text
Change remote DTO fields freely.
```

Correct:

```text
Version contracts or preserve backward compatibility.
```

### Mistake 9: Forwarding all headers

Wrong:

```text
Copy every incoming header to downstream.
```

Correct:

```text
Propagate only safe required headers.
```

## 30. Interview-Ready Explanation

If interviewer asks:

```text
How do you use OpenFeign in microservices?
```

Strong answer:

```text
OpenFeign is a declarative HTTP client. I define a Java interface annotated with
@FeignClient and map methods to remote HTTP endpoints using Spring MVC
annotations. Spring creates a proxy implementation, so service code can call the
remote API like a method. But I always treat that method as a network boundary,
not a local call. That means I configure explicit connect/read timeouts, map
remote HTTP errors through an ErrorDecoder, add correlation ID propagation,
protect important calls with circuit breaker and limited retry, and design
fallback behavior based on business criticality. For service discovery, Feign can
use the service name with Eureka and Spring Cloud LoadBalancer. I avoid exposing
JPA entities as Feign contracts and test the client with mocks plus integration
or contract tests. For security-sensitive calls like abuse checks, I usually fail
closed instead of returning a success fallback.
```

Why this is strong:

```text
1. Explains Feign clearly.
2. Mentions proxy and declarative HTTP.
3. Shows distributed-system awareness.
4. Mentions timeout, retry, circuit breaker.
5. Explains Eureka integration.
6. Discusses fallback as business decision.
7. Avoids entity coupling.
8. Mentions testing and observability.
```

Senior one-liner:

```text
Feign makes HTTP calls look like Java methods, but production design must still treat them as unreliable remote calls.
```
